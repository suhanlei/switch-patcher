# Switch Patcher 开发沟通记录

日期：2026-05-27

## 一、打补丁流程梳理及SSH登录次数分析

### 1.1 完整6步流程

| 步骤 | 动作 | SSH登录次数 |
|------|------|------------|
| 1. check_scp | TCP探测 → SSH登录检查SCP/SFTP状态 | 1 |
| 2. open_scp | SSH登录执行SCP/SFTP使能命令 → 保存配置 | 1 |
| 3. recheck_scp | SSH登录再次确认SCP/SFTP已开启 | 1 |
| 4. upload | paramiko SFTP上传补丁文件 + SSH登录验证文件完整性 | 2 |
| 5. activate | SSH预检查+激活+后检查（同一连接） | 1 |
| 6. finalize | 汇总报告 | 0 |

**完整流程每台交换机登录6次**（5次netmiko SSH + 1次paramiko SFTP Transport）

断点续跑减少次数：
- SCP已开启 → 跳过步骤2/3，减少2次
- `dry_run` 模式 → 步骤5只做预检查，减少0次（同一连接内完成）
- 已上传成功 → 跳过步骤4，减少2次
- 已激活成功 → 跳过步骤5，减少1次
- 补丁已激活 → 步骤5预检查检测到后跳过，减少0次（同一连接内完成）

**关键设计点**：步骤5的预检查、激活、后检查在同一SSH连接中完成。激活过程中设备持续回显进度，SSH不会因空闲超时断开，`read_until_prompt`只在连续120秒无数据时才超时。

## 二、本次代码变更总结（未提交）

### 2.1 device_worker.py — 统一命令发送 + 全局Y/N自动回复

**变更前**：3套不同的命令发送逻辑
- `_send_command_interactive()` — 交互式Y/N确认（逐条在YAML里配置expect_pattern/auto_reply）
- `_send_command_wait_progress()` — 进度等待（锐捷3步）
- `conn.send_config_set()` — 普通配置命令

**变更后**：统一为 `_send_cmd()` 一个函数
- 全局检测 `[Y/N]:` 提示并自动回复Y，无需在YAML里逐条配置
- `wait_progress=True` 时持续等待直到设备提示符出现
- 自动去掉设备回显的命令本身（`_strip_echo`）
- 优先使用 `send_command_timing`（兼容H3C Comware）
- `_send_command_interactive` 和 `_send_command_wait_progress` 两个函数删除

**影响**：YAML模板中的 `expect_pattern` / `auto_reply` 字段不再被使用（华为huawei.yaml仍保留但实际不生效），所有Y/N确认由全局正则 `_CONFIRM_PATTERN` 统一处理。

### 2.2 device_worker.py — config_mode_required 控制

**变更前**：步骤5激活时无条件 `conn.config_mode()` 进入配置模式

**变更后**：根据YAML模板 `config_mode_required` 字段决定是否进入config模式
- H3C: `false` — `install activate/commit` 在用户视图执行
- 华为: `false` — `patch load` 在用户视图执行
- 锐捷: `false` — `upgrade/patch active/patch running` 在用户视图执行

**原因**：三大厂商的补丁命令都在用户视图执行，之前强制进config模式会导致命令执行失败。

### 2.3 connection.py — H3C大缓冲区实现方式变更

**变更前**：通过netmiko参数 `recv_buffer_size` 传递（netmiko 4.2.0已不支持此参数）

**变更后**：连接建立后，通过paramiko Transport设置接收窗口
- `_set_transport_window()` — 通过 `ssh_client.get_transport()` 获取Transport对象
- 设置 `default_window_size`、`REKEY_BYTES`、`REKEY_PACKETS` 三个参数
- 备用路径：通过channel获取Transport

**新增连接参数**：`allow_agent: False`、`ssh_strict: False`

### 2.4 health_check.py — 补丁后检查逻辑重写

**变更前**：统一执行 `display patch information`，简单检查文件名

**变更后**：按厂商分支处理
- H3C: `display install active` → `display patch information` 兜底
- 华为: `display patch-information` 检查 State=Running
- 其他: 通用 `display patch information` 检查文件名

**原因**：Comware V7/V9 不存在 `display patch information` 命令（那是V5的命令），H3C应使用 `display install active` 验证。

## 三、合并预检查与激活步骤（减少SSH登录次数）

### 3.1 问题分析

步骤5 `step_activate` 原本使用3次SSH连接：预检查 → 激活 → 后检查。其中预检查单独占1次连接，但实际价值有限：

1. **H3C 错误检测无效**：`H3C_PATCH_ERROR_PATTERNS`（cannot be activated again / not compliant / does not exist）是从 `display install active` 的输出中匹配，但这些错误文本是**执行激活命令时**的报错，不会出现在 display 查询结果中——预检查根本检测不到
2. **patch_id_pattern 匹配错误**：`'flash:/([^\s]+)'` 会匹配到第一个文件（BOOT.bin），而非补丁文件，`patch_now` 写入 Excel 的值不对
3. **预检查唯一实际价值**：获取当前补丁版本（信息记录）+ 判断目标补丁是否已激活

### 3.2 改动内容

| 文件 | 改动 |
|------|------|
| `vendor_templates/h3c.yaml` | `patch_id_pattern` 从 `'flash:/([^\s]+)'` 改为 `'(?i)flash:/(\S*[Hh][Ss]?\d+\S*\.bin)'`，匹配含H/HS标识的补丁文件，不区分大小写 |
| `vendor_templates/h3c.yaml` | 修正误编辑 `display 可以` → `display install committed` |
| `switch_patcher/health_check.py` | 删除 `H3C_PATCH_ERROR_PATTERNS` 和 `HealthStatus.patch_errors` 字段 |
| `switch_patcher/health_check.py` | 新增 `HealthStatus.patch_already_active` 字段 |
| `switch_patcher/health_check.py` | `patch_already_active` 检测改为 `basename.lower() in output.lower()`，避免大小写不一致导致误判 |
| `switch_patcher/health_check.py` | `run_health_checks` 新增 `expected_patch_file` 参数，从输出中检测目标补丁是否已激活 |
| `switch_patcher/device_worker.py` | 预检查和激活合并为同一SSH连接，从3次登录减为2次 |

### 3.3 效果

- 完整流程每台交换机登录次数从 **8次 → 6次**
- 步骤5从3次SSH连接减为1次（预检查+激活+后检查同一连接）
- 新增"补丁已激活"检测：预检查中如果目标补丁已在激活列表中，写 `SKIP-already active` 并跳过
- `dry_run` 模式不再额外省SSH连接（预检查和后检查在同一连接内完成）

### 3.4 补丁命名兼容性修复

H3C 补丁文件名大小写不统一，且命名格式有两种：

| Comware 版本 | 补丁文件名示例 | 标识 |
|---|---|---|
| V7 | `s9850_6850-cmw710-system-r6555p01h35.bin` | `H`（无S） |
| V9 | `S9857_9827-CMW910-SYSTEM-R9316HS04.bin` | `HS` |

修正：
1. `patch_id_pattern` 正则加 `(?i)` 不区分大小写，`S?` 让 S 可选
2. `patch_already_active` 检测用 `lower()` 比对，避免大小写不一致

## 四、H3C回退逻辑修正

### 4.1 问题

初始回退命令使用 `install deactivate`，但 H3C 可以直接 `install activate` 旧补丁回退，不需要先 deactivate 新补丁。

### 4.2 修正

回退命令改为：activate 旧补丁 → commit → 删除新补丁文件

| 文件 | 改动 |
|------|------|
| `vendor_templates/h3c.yaml` | 回退命令改为 `install activate patch flash:/{patch_old} all` → `install commit` → `delete` |
| `switch_patcher/vendor_profiles.py` | `format_command` 新增 `patch_old` 占位符 |
| `switch_patcher/device_worker.py` | `_generate_rollback` 传入 `patch_now`（旧补丁名），回退文件头部标注旧补丁信息 |

### 4.3 注意事项

- 回退前提：旧补丁文件必须仍在设备 flash 上，否则需先 SFTP 上传
- 回退文件头部已标注前置检查步骤：`dir flash:/{patch_old}` 确认文件存在，不存在则先上传
